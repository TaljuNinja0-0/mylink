# 마이링크 (MyLink) - 제품 요구사항 정의서 (PRD)

## 1. 프로젝트 개요
- **프로젝트명**: 마이링크 (MyLink)
- **목적**: 사용자가 자신의 여러 소셜 미디어, 포트폴리오, 웹사이트 등의 링크를 하나의 통합된 페이지에서 관리하고 공유할 수 있도록 지원하는 심플하고 직관적인 링크 관리 플랫폼 (Linktree 클론).
- **대상 사용자**: 개발자 및 크리에이터 중심 (포트폴리오, 이력서, 깃허브, 소셜 미디어 링크 등을 한 곳에서 관리하고자 하는 사용자)
- **기술 스택 및 디자인**: Firebase (소셜 로그인 등), shadcn/ui 디자인 시스템 적용 (구체적인 디자인 컨셉은 차후 구체화)

## 2. 핵심 기능 목록

### 필수 기능 (MVP - Minimum Viable Product)
- **사용자 인증**: Google 소셜 로그인 기능만 지원 (Firebase Authentication 활용)
- **고유 URL 생성**: 사용자의 지메일 아이디 앞부분을 추출하여 설정된 `displayName`을 기반으로 고유 URL 제공 (예: `mylink.com/emailid`)
- **링크 관리 (CRUD 및 통계)**:
  - 새로운 링크 추가 (제목, URL 필수)
  - 추가된 링크 아이템은 Google Favicon API를 통해 대상 링크의 파비콘을 아이콘으로 자동 추출하여 적용
  - 인라인 편집(Inline Editing) 방식으로 기존 링크 및 프로필 정보 수정 지원 (순서 변경/토글 제거)
  - 개별 링크 클릭 조회수 통계 노출
- **프로필 설정**:
  - Google 계정의 프로필 이미지(photoURL) 자동 연동 표시 (별도 이미지 업로드/변경 불가)
  - 실제 사용자 이름(username), 이메일 주소 표기, 짧은 자기소개(Bio) 설정 및 인라인 편집 지원
- **기본 랜딩 페이지 뷰**: 방문자가 볼 수 있는 최종 사용자(End-user) 페이지 제공

## 3. 각 기능 및 모델링 상세 설명

### 3.1 사용자 인증 및 계정 관리
- Firebase 기반의 **Google 소셜 로그인**만 유일한 인증 수단으로 사용합니다. (다른 통합 수단 미지원)
- 로그인 시 구글 계정에서 **이메일(email)** 정보를 수집하여 프로필 화면에 활용합니다.
- 계정 생성 시 퍼블릭 랜딩 페이지 고유 URL 식별자 역할을 하는 **`displayName`** 필드를 설정하며, 이 값은 사용자의 구글 지메일 아이디 앞부분(예: user@gmail.com -> user)을 파싱하여 자동 부여합니다. 이를 바탕으로 `mylink.com/displayName` 경로가 생성됩니다.

### 3.2 링크 관리 시스템 (대시보드)
- 로그인한 사용자는 관리 대시보드에서 자신의 링크 목록을 직관적으로 확인할 수 있습니다.
- **링크 추가**: "새 링크 추가" 버튼을 통해 즉시 Title과 URL 입력 폼을 생성합니다.
- **파비콘 및 클릭수 연동**: Google Favicon API를 호출하여 링크의 아이콘을 표시하고, 클릭이 발생할 때마다 해당 링크의 클릭 조회수를 누적하여 보여줍니다.

### 3.3 사용자 프로필 설정
- 랜딩 페이지의 가장 상단에 노출되는 프로필 사진(구글 계정 연동), 실제 사용자 이름(username), 이메일 주소, 자기소개를 관리합니다.
- 프로필 사진은 1:1 비율로 크롭 및 리사이징되어 원형 또는 둥근 사각형 형태로 표시되며, 사용자 직접 업로드는 불가합니다.
- 링크 제목과 URL, 사용자 이름(username) 및 소개글은 클릭 즉시 내용 수정이 가능한 **인라인 편집(Inline editing)** 방식으로 구현합니다. (이메일은 구글 계정 연동으로 고정)

### 3.4 랜딩 페이지 접속 경험
- 사용자의 `displayName` 경로를 통해 접속한 퍼블릭 페이지는 반응형 웹으로 구현됩니다. (별도 모바일 뷰어 미제공)
- 링크 클릭 시 외부 브라우저나 새 탭으로 전환(`target="_blank"`)되며, 이때 클릭 카운트가 증가합니다.

### 3.5 데이터베이스 모델링 (Firebase Firestore)
- 관리의 효율성을 위해 Firestore를 사용하며, 각 사용자의 구체적인 링크 데이터는 하위의 **서브 컬렉션(Sub-collection)**으로 분리하여 관리합니다.

**Firestore 스키마 설계**
```text
Collection: users
  Document: {uid} (Firebase Auth UID)
    - displayName (string): 접속을 위한 랜딩 페이지 고유 URL 식별자 (구글 지메일 아이디의 앞부분을 활용해 자동 생성)
    - username (string): 화면에 노출되는 실제 사용자 이름
    - email (string): 사용자 이메일 주소 (Google 계정 정보 연동)
    - bio (string): 짧은 자기소개
    - photoURL (string): 구글 연동 프로필 사진 (업로드 불가)
    - createdAt (timestamp): 가입 일시

    Sub-collection: links
      Document: {linkId} (고유 문서 ID)
        - title (string): 링크 제목
        - url (string): 목적지 URL
        - clickCount (number): 개별 링크 누적 클릭 횟수
        - createdAt (timestamp): 생성 일시
        ※ 파비콘 URL은 `url` 도메인을 추출하여 Google Favicon API에서 실시간으로 렌더링
```
